4. 遇到的问题与解决方法
问题一：发布信息成功后，列表页面无法展示新发布内容
问题描述
原有发布表单提交逻辑仅弹出成功提示并跳转首页，没有任何数据存储逻辑。所有业务列表页面依赖静态 mock 固定数组渲染，无法识别用户新增发布数据，导致发布功能完全失效。
解决方法
新建 Pinia 仓库 itemStore，将静态模拟数据转为响应式数据；为四类发布页面分别编写数据新增方法，提交表单时将新数据推入全局状态；所有列表页面改用 computed 监听仓库数据，实现发布即刷新、实时渲染的动态数据联动效果。
问题二：部分业务页面图片加载失败、图片空白
问题描述
项目中原使用 Unsplash 图片 ID 部分已失效、资源被删除，返回 404 错误，导致钥匙、充电宝、校园物品等业务图片无法加载，页面视觉残缺。
解决方法
统一替换图片资源服务，放弃不稳定的 Unsplash ID 链接，采用 picsum.photos 带关键词种子的稳定图片链接，根据业务场景匹配对应图片，保证所有页面图片永久可访问，彻底解决图片空白问题。
5. 今日反思
页面骨架、路由导航和公共布局是项目的地基工程。今天搭建的 AppLayout + AppHeader + AppNav 三位一体的布局体系，让后续所有页面开发都有了统一的容器和导航框架——开发者只需要关注每个页面内部的业务逻辑，不用再重复处理顶部导航和页面结构。Vue Router 的配置让 8 个页面形成了完整的跳转闭环，从首页浏览到分类筛选，再到发布信息和个人中心，用户的核心路径已经跑通。同时，今天对 AI 生成代码的"删减"实践也让我意识到：AI 擅长生成完备的方案，但开发者必须根据当前阶段的需求做减法，避免过早优化和过度工程化。明天可以在此基础上，优先填充首页和分类列表页的真实数据和业务逻辑。
